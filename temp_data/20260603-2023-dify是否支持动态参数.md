# Dify 是否支持动态参数 —— 从「设备下拉调后端」到源码级排查全记录

> **主题**：Dify 工作流工具插件能否让参数下拉选项由我们自己的 Spring Boot 接口动态返回？  
> **结论先行**：**支持**，参数类型为 `dynamic-select`，必须在插件 Python 中实现 `_fetch_parameter_options`；浏览器**不会**直接请求 Spring。在 Dify 1.12.x 工作流场景下，还存在**前端未向第二个 ToolForm 传递 provider 上下文**的缺陷，需将动态参数设为 `form: llm` 或修补 Dify 前端。

## 1. 这篇博客要回答什么

在前几篇博客里，我们已经做到：

- 插件通过 `iot_device_http.difypkg` 安装到远程 Dify；
- 工作流里「获取设备列表」等工具能调通本地 Spring，返回 3 台模拟设备；
- `generic_http` 用路径加方法访问后端任意 REST 接口。

随后连续出现的一类需求是：

> **画布上配置工具节点时，某个参数能不能变成下拉框，选项来自我们的 Java 接口，而不是写死在 YAML 里？**

这就是本文所说的**动态参数**。在 Dify 插件体系里，它对应 SDK 的 **`dynamic-select`** 参数类型，以及 Python 侧的 **`_fetch_parameter_options`** 钩子。

本文用流水账记录：我们怎么问、怎么试、怎么错、怎么对照 `dify/` 目录下的前端源码定位根因，以及最终怎样才能真正「调用后端接口」填充下拉。

## 2. 流水账：十几个问题如何汇聚到「动态参数」

下面按时间线归纳对话与实验中的主要问题，便于和 FAQ 博客对照。

| 序号 | 问题摘要 | 当时结论 |
|------|----------|----------|
| 1 | difypkg 里到底有什么 | 源码、YAML、Python、依赖清单，不含 Spring |
| 2 | Python 插件项目职责 | 桥接 Dify 与 Spring，不做业务 UI |
| 3 | 每个后端接口都要写一个工具吗 | 不必，可用 generic_http |
| 4 | 插件升级会不会冲掉工作流 | 工具名稳定则影响小 |
| 5 | 能否用 Go 写插件 | 官方以 Python 为主 |
| 6 | 一个通用 HTTP 能否覆盖全部 API | 能，运行期灵活，**配置期无动态下拉** |
| 7 | 插件重名怎么办 | 改 author 或 name |
| 8 | 多参数怎么传 | form llm 与 form form 分工 |
| 9 | **能否参数下拉来自后端** | **能，但要 dynamic-select 加 Python，不是只改 Spring** |
| 10 | 只改 generic_http.yaml 够不够 | **不够**，要 Python 钩子 |
| 11 | 不要动 generic_http，新建工具 | 新增 dynamic_device_query |
| 12 | 打包安装后仍不能选设备 | 前端 form 分区 bug 加 SDK 280 加凭证网络 |
| 13 | 对照 dify 源码再分析 | 确认 panel 第二个 ToolForm 未传 provider |

核心认知变化：

1. 一开始以为「Spring 加一个列表接口，Dify 会自动来拉」——**错**。  
2. 然后以为「YAML 里写 `type: dynamic-select` 就行」——**半对**，还要 Python。  
3. 再以为「插件装好了就一定能下拉」——**错**，1.12 前端对 `form: form` 的动态参数可能**从不发请求**。  
4. 最后对照源码：**插件写对 + Spring 可达 + 参数放在 form llm 或修前端**，三者缺一不可。

## 4. Dify 官方能力：dynamic-select 是什么

### 4.1 Schema 层

在 `dify_plugin>=0.9.0` 的 `ToolParameter.ToolParameterType` 中，包含 `DYNAMIC_SELECT`（YAML 里写作 `dynamic-select`）。

工具参数 YAML 示例：

```yaml
parameters:
  - name: device_id
    type: dynamic-select
    required: true
    form: llm
    label:
      zh_Hans: 设备
```

### 4.2 Python 层

工具类需实现：

```python
def _fetch_parameter_options(self, parameter: str) -> list[ParameterOption]:
    ...
```

返回 `ParameterOption(value=..., label=I18nObject(...))` 列表。

### 4.3 与已废弃的 runtime_parameters 区分

SDK 中 `_get_runtime_parameters` 标记为 deprecated，提示由 dynamic-select 替代。本文**未**走该老路径，专注 `dynamic-select`。

### 4.4 与 Trigger 插件的对比

Trigger 订阅配置里广泛使用 `dynamic-select`，且前端有 `useTriggerPluginDynamicOptions` 专用逻辑。纯 **Tool** 插件在工作流节点里走 `useFetchDynamicOptions`，路径相同但挂载组件不同，**Tool 的 panel 存在下文所述 bug**。

## 6. 实现记录：Spring 新接口与插件新工具

用户要求：**不修改** `generic_http`，新建接口、新 py、新 yaml。

### 6.1 Spring：DifyPluginSelectController

```java
@RestController
@RequestMapping("/api/dify-plugin")
public class DifyPluginSelectController {

    @GetMapping("/device-select-options")
    public ResponseEntity<List<SelectOption>> listDeviceSelectOptions() {
        log.info("收到获取设备下拉选项请求");
        return ResponseEntity.ok(difyPluginSelectService.listDeviceSelectOptions());
    }
}
```

`DifyPluginSelectService` 从现有 `DeviceService.listDevices()` 生成 `value` 与 `label`，不改动原有 `DeviceController` 五个 API。

### 6.2 插件：dynamic_device_query

**注册**（`provider/iot_device_plugin.yaml`）：

```yaml
tools:
  - tools/dynamic_device_query.yaml
```

**入口补丁**（`main.py`）：

```python
from plugin_bootstrap import apply_sdk_patches
apply_sdk_patches()
from dify_plugin import Plugin, DifyPluginEnv
plugin = Plugin(DifyPluginEnv())
plugin.run()
```

**SDK 280 兼容**（`plugin_bootstrap.py`）：纯 Tool 插件拉取 dynamic-select 时，SDK 会先查 Trigger，`TriggerFactory._get_entry` 对非 Trigger 提供方抛 `ValueError`，导致永远走不到 `Tool._fetch_parameter_options`。补丁在 `get_trigger_event_handler_safely` 捕获后返回 `None`。

**requirements.txt：**

```
dify_plugin>=0.9.0,<1.0.0
requests>=2.31.0
python-dotenv>=1.0.0
```

## 7. 第一次失败：只改后端或只改 YAML

### 7.1 测试数据 A — 假设「Spring 加接口即可」

| 项 | 内容 |
|----|------|
| 操作 | 仅新增 `GET /api/dify-plugin/device-select-options`，不改插件 |
| 现象 | 工作流节点里仍无设备下拉；curl 本地 200 |
| 运行流程 | 画布渲染参数 → 只读插件 manifest 静态 schema → **无代码路径请求 Spring** |
| 结论 | **配置期下拉与 Spring 无直接连接** |

### 7.2 测试数据 B — 在 generic_http 把 path 改成 dynamic-select

| 项 | 内容 |
|----|------|
| 操作 | 尝试改 `generic_http.yaml` 的 path 为 dynamic-select |
| 现象 | 用户随后要求回退，保留 generic_http 通用能力 |
| 运行流程 | 即便保留，也需在 `generic_http.py` 实现 `_fetch_parameter_options`；且 generic_http 的 path 为 **form llm 的 string**，本意是运行期拼 URL |
| 结论 | 应用**新工具**承载动态设备查询，而不是污染 generic_http |

## 8. 第二次失败：dynamic-select 加 form form

### 8.1 初版 YAML 参数分区

最初 `device_id` 配置为：

```yaml
  - name: device_id
    type: dynamic-select
    form: form
```

`query_type` 等为 `form: form` 的静态 `select`。

### 8.2 问题现象 用户侧

- 重装 `iot_device_http.difypkg` 后，打开「动态设备查询」节点；
- **设备**下拉为空，或不可选；
- 浏览器 Network **没有** `dynamic-options` 请求；
- plugin-daemon 日志 **没有** `[dynamic_device_query] _fetch_parameter_options called`；
- 用户以为「没调用后端」——实际上 **Dify 前端未调用插件**，更未到 Spring。

### 8.3 测试数据 C — device_id 在「设置」区

| 步骤 | 系统行为 |
|------|----------|
| 1 | `use-config.ts` 把 `form !== llm` 的参数放进 `toolSettingSchema` |
| 2 | `panel.tsx` 在「设置」区渲染第二个 `ToolForm` |
| 3 | 第二个 `ToolForm` **未传** `currentProvider` `currentTool` |
| 4 | `form-input-item.tsx` 中 `useEffect` 判断 `currentTool && currentProvider` 失败 |
| 5 | 不调用 `useFetchDynamicOptions` |
| 6 | Python 与 Spring 均不被触发 |

这与 GitHub [dify#36518](https://github.com/langgenius/dify/issues/36518) 描述一致：Tool 插件 `form: form` 的 dynamic-select 在 1.6.0 之后常失效；修复 PR [dify#36743](https://github.com/langgenius/dify/pull/36743) 给第二个 ToolForm 补传 provider，**我们检出的 main 分支 panel 仍未合并该修复**。

## 9. 对照 Dify 源码：断点在哪里

工作区 `dify/web` 关键片段。

### 9.1 参数拆分 use-config.ts

```typescript
const toolInputVarSchema = useMemo(() => {
  return formSchemas.filter((item: any) => item.form === 'llm')
}, [formSchemas])
const toolSettingSchema = useMemo(() => {
  return formSchemas.filter((item: any) => item.form !== 'llm')
}, [formSchemas])
```

注释写明：`tool_configurations` 对应 form form，`tool_parameters` 对应 form llm。

### 9.2 双 ToolForm panel.tsx

第一个 ToolForm 传入：

```tsx
currentProvider={currCollection}
currentTool={currTool}
```

第二个 ToolForm **没有**上述 props：

```tsx
<ToolForm
  readOnly={readOnly}
  nodeId={id}
  schema={toolSettingSchema as any}
  value={toolSettingValue}
  onChange={setToolSettingValue}
/>
```

### 9.3 动态拉取 form-input-item.tsx

```typescript
useEffect(() => {
  const fetchPanelDynamicOptions = async () => {
    if (isDynamicSelect && currentTool && currentProvider
        && (providerType === PluginCategoryEnum.tool || ...)) {
      const data = await fetchDynamicOptions()
      setToolsOptions(data?.options || [])
    }
  }
  fetchPanelDynamicOptions()
}, [...])
```

### 9.4 前端请求的 API use-plugins.ts

```typescript
mutationFn: () => get('/workspaces/current/plugin/parameters/dynamic-options', {
  params: { plugin_id, provider, action, parameter, provider_type, ...extra },
}),
```

官方 dify-api（main 分支）对应 `PluginParameterService.get_dynamic_select_options`，再通过 `DynamicSelectClient` POST 到 daemon：

`plugin/{tenant_id}/dispatch/dynamic_select/fetch_parameter_options`

### 9.5 排查时应在哪一层看日志

| 层级 | 看什么 |
|------|--------|
| 浏览器 | 是否有 `dynamic-options`，query 含 plugin_id provider action parameter |
| dify-api | 是否 400 plugin_error，凭证是否解密成功 |
| plugin-daemon | 是否 dispatch dynamic_select |
| 插件 stderr | `[dynamic_device_query] _fetch_parameter_options called` |
| Spring | `收到获取设备下拉选项请求` 或 access log |

## 10. 完整调用链与每条测试数据的运行流程

### 10.1 修复后配置 device_id 使用 form llm

当前 `dynamic_device_query.yaml` 片段：

```yaml
  - name: device_id
    type: dynamic-select
    form: llm
```

设备参数出现在节点 **「输入变量」** 区域，走第一个 ToolForm，具备 provider 上下文。

### 10.2 测试数据 1 — 下拉加载 device_001

| 字段 | 值 |
|------|-----|
| 场景 | 用户打开节点，等待下拉 |
| device_id | device_001 |
| query_type | status 默认 |

**运行流程：**

1. 浏览器挂载 `FormInputItem`，`isDynamicSelect=true`，`currentTool`=`dynamic_device_query`，`currentProvider` 含 `plugin_id`。  
2. GET `.../plugin/parameters/dynamic-options?plugin_id=...&provider=...&action=dynamic_device_query&parameter=device_id&provider_type=tool`。  
3. api 取租户凭证 `spring_service_url`，daemon 调插件。  
4. `DynamicDeviceQueryTool._fetch_parameter_options("device_id")` 执行，stderr 打印 called。  
5. 插件 GET `http://局域网IP:8080/api/dify-plugin/device-select-options`。  
6. 返回 3 条 Option，前端展示「客厅温度传感器 (device_001)」等。  
7. 用户选中 device_001 后点运行，走 `_invoke`，GET `/api/devices/device_001/status`。  

**期望 Spring 日志：** 先下拉接口一次，运行期 status 接口一次。

### 10.3 测试数据 2 — 下拉加载 device_002 query_type detail

| 字段 | 值 |
|------|-----|
| device_id | device_002 |
| query_type | detail |

**运行流程：**

- 步骤 1～6 同上，选项中 label 含「卧室智能灯泡」。  
- `_invoke` 中 `query_type == "detail"`，请求 `GET /api/devices/device_002`。  
- 返回 Device JSON，插件 `create_text_message` 与 `create_json_message`。

### 10.4 测试数据 3 — device_003 历史数据

| 字段 | 值 |
|------|-----|
| device_id | device_003 |
| query_type | data |
| metric | temperature |
| limit | 5 |

**运行流程：**

- 下拉仍走 `_fetch_parameter_options`。  
- `_invoke` 请求 `GET /api/devices/device_003/data?metric=temperature&limit=5`。  
- `DeviceService` 生成模拟时序点。  

**说明**：device_003 在 mock 中为 offline，status 仍可读，与下拉无关。

### 10.5 测试数据 4 — spring_service_url 填 localhost 的失败路径

| 字段 | 值 |
|------|-----|
| 凭证 | http://localhost:8080 |

**现象：**

- 下拉或运行报连接失败；  
- 插件日志 `拉取设备选项失败` 或 `连接失败：无法访问`。  

**运行流程：**

- 插件进程在 **K8s pod** 内，`localhost` 指向 pod 自身，不是宿主机 Spring。  

**修复：** 凭证改为宿主机局域网 IP，如 `http://192.168.x.x:8080`。

### 10.6 测试数据 5 — 兜底 list_devices 加变量引用

| 字段 | 值 |
|------|-----|
| 工具链 | list_devices → 下游 get_device_status |

**运行流程：**

1. list_devices `_invoke` → Spring `GET /api/devices` → 文本列出 device_001 等。  
2. 下游 `device_id` 用 **变量引用** 上游输出，**不经过 dynamic-select**。  
3. 配置期无下拉，**运行期一定调 Spring**。

### 13.2 dynamic_device_query.py 全文

```python
import json
import sys
from typing import Any, Generator

import requests
from dify_plugin import Tool
from dify_plugin.entities import I18nObject, ParameterOption
from dify_plugin.entities.tool import ToolInvokeMessage

DEVICE_SELECT_OPTIONS_PATH = "/api/dify-plugin/device-select-options"


class DynamicDeviceQueryTool(Tool):
    def _spring_url(self) -> str:
        return self.runtime.credentials.get("spring_service_url", "").rstrip("/")

    def _request_headers(self) -> dict[str, str]:
        headers = {"Accept": "application/json"}
        api_token = self.runtime.credentials.get("api_token", "")
        if api_token:
            headers["Authorization"] = f"Bearer {api_token}"
        return headers

    def _fetch_parameter_options(self, parameter: str) -> list[ParameterOption]:
        print(
            f"[dynamic_device_query] _fetch_parameter_options called: parameter={parameter}",
            flush=True,
            file=sys.stderr,
        )
        if parameter != "device_id":
            return []

        spring_url = self._spring_url()
        if not spring_url:
            print(
                "[dynamic_device_query] spring_service_url 为空，无法拉取设备列表",
                flush=True,
                file=sys.stderr,
            )
            return []

        url = f"{spring_url}{DEVICE_SELECT_OPTIONS_PATH}"
        try:
            response = requests.get(
                url,
                headers=self._request_headers(),
                timeout=15,
            )
            response.raise_for_status()
            items = response.json()
            print(
                f"[dynamic_device_query] GET {url} -> {len(items)} options",
                flush=True,
                file=sys.stderr,
            )
        except Exception as e:
            print(
                f"[dynamic_device_query] 拉取设备选项失败: {url} error={e}",
                flush=True,
                file=sys.stderr,
            )
            return []

        options: list[ParameterOption] = []
        for item in items:
            value = str(item.get("value", ""))
            label_text = str(item.get("label", value))
            if not value:
                continue
            options.append(
                ParameterOption(
                    value=value,
                    label=I18nObject(en_US=label_text, zh_Hans=label_text),
                )
            )
        return options

    def _invoke(
        self, tool_parameters: dict[str, Any]
    ) -> Generator[ToolInvokeMessage, None, None]:
        spring_url = self._spring_url()
        api_token = self.runtime.credentials.get("api_token", "")
        device_id = (tool_parameters.get("device_id") or "").strip()
        query_type = (tool_parameters.get("query_type") or "status").strip()
        metric = tool_parameters.get("metric", "temperature") or "temperature"
        limit = int(tool_parameters.get("limit", 5) or 5)

        if not device_id:
            yield self.create_text_message("错误：请从下拉框选择设备（device_id）")
            return

        headers = {"Content-Type": "application/json", "Accept": "application/json"}
        if api_token:
            headers["Authorization"] = f"Bearer {api_token}"

        if query_type == "detail":
            url = f"{spring_url}/api/devices/{device_id}"
        elif query_type == "status":
            url = f"{spring_url}/api/devices/{device_id}/status"
        elif query_type == "data":
            url = f"{spring_url}/api/devices/{device_id}/data"
        else:
            yield self.create_text_message(f"错误：不支持的查询类型 {query_type}")
            return

        params = {"metric": metric, "limit": limit} if query_type == "data" else None

        try:
            response = requests.get(
                url, headers=headers, params=params, timeout=15
            )
            response.raise_for_status()
            data = response.json()
        except requests.exceptions.HTTPError as e:
            if e.response is not None and e.response.status_code == 404:
                yield self.create_text_message(f"设备 {device_id} 不存在")
            else:
                code = e.response.status_code if e.response is not None else "?"
                yield self.create_text_message(f"请求失败: HTTP {code}")
            return
        except requests.exceptions.ConnectionError:
            yield self.create_text_message(
                f"连接失败：无法访问 {url}\n请确认 Spring 服务已启动且插件凭证地址正确。"
            )
            return
        except Exception as e:
            yield self.create_text_message(f"请求失败: {str(e)}")
            return

        title = {"detail": "设备详情", "status": "设备状态", "data": "历史数据"}.get(
            query_type, query_type
        )
        formatted = json.dumps(data, ensure_ascii=False, indent=2)
        yield self.create_text_message(
            f"【{title}】设备 {device_id}\n请求: GET {url}\n\n{formatted}"
        )
        yield self.create_json_message(data)
```

### 13.3 plugin_bootstrap.py 全文

```python
"""
dify_plugin 0.9.x：纯 Tool 插件在拉取 dynamic-select 时会误查 Trigger 并抛错，
导致永远不会执行 Tool._fetch_parameter_options。此处做兼容补丁。
参见 https://github.com/langgenius/dify-plugin-sdks/issues/280
"""

from dify_plugin.core.trigger_factory import TriggerFactory

_original_get_trigger_event_handler_safely = (
    TriggerFactory.get_trigger_event_handler_safely
)


def _safe_get_trigger_event_handler_safely(
    self, provider_name: str, event: str, runtime
):
    try:
        return _original_get_trigger_event_handler_safely(
            self, provider_name, event, runtime
        )
    except ValueError:
        return None


def apply_sdk_patches() -> None:
    TriggerFactory.get_trigger_event_handler_safely = (
        _safe_get_trigger_event_handler_safely
    )
```

## 14. 与前几篇博客的衔接：动态参数在整条链路中的位置

### 14.1 前三篇已经打通的部分

| 博客 | 解决的问题 | 与动态参数的关系 |
|------|------------|------------------|
| 1354 插件接口怎么连 | Windows 到 K8s daemon 的 TCP | 没有 daemon 则插件进程不存在，dynamic-select 无入口 |
| 1425 本地 debug | REMOTE_INSTALL_HOST 等 | debug 模式可在本机看 stderr，适合验证 `_fetch_parameter_options` 是否被调用 |
| 1548 difypkg 安装 | ConnectionClosedError 与正式包 | 正式安装后由 daemon 拉起的容器必须能访问 Spring 局域网地址 |

FAQ 1848 中记录：工作流三节点全部 succeeded，说明 **运行期 invoke 链路** 已通。动态参数是 **配置期** 的独立链路，运行期成功不代表下拉会成功。

### 14.2 凭证校验为何「通过」仍拉不到下拉

`iot_device_plugin.py` 的 `_validate_credentials` **只校验 URL 格式**：

```python
def _validate_credentials(self, credentials: dict[str, Any]) -> None:
    spring_url = credentials.get("spring_service_url", "").rstrip("/")
    if not spring_url:
        raise ToolProviderCredentialValidationError("Spring服务地址不能为空")
    if not spring_url.startswith("http"):
        raise ToolProviderCredentialValidationError(
            f"Spring服务地址必须以http开头: {spring_url}"
        )
```

安装插件时打印示例：

```
[DEBUG] _validate_credentials 被调用! credentials={'spring_service_url': 'http://192.168.1.100:8080', ...}
[DEBUG] 凭证格式校验通过: spring_url=http://192.168.1.100:8080
```

这 **不会** 请求 Spring。因此安装成功与下拉可用之间没有因果关系。下拉阶段 api 会从数据库取出同一凭证交给插件；若此时填的是 `localhost`，安装与下拉都会「逻辑上」有地址，但容器内访问必失败。

## 15. form llm 与 form form：官方语义与我们的参数布局

Dify 工具文档说明（概括）：

- **form llm**：在工作流中属于「输入变量」，可填常量，也可接上游变量；Agent 场景下还可由模型推断。
- **form form**：属于工具「设置」，用户在节点面板预先填好，每次运行复用。

我们的 `dynamic_device_query` 最终布局：

| 参数 | type | form | 面板区域 | 是否触发 dynamic-options |
|------|------|------|----------|-------------------------|
| device_id | dynamic-select | llm | 输入变量 | 是 第一个 ToolForm |
| query_type | select | form | 设置 | 否 静态 options |
| metric | string | form | 设置 | 否 |
| limit | number | form | 设置 | 否 |

**经验规则**：凡是要「打开节点就从后端拉下拉」的 Tool 参数，在 Dify 1.12 工作流里应优先放在 **form llm**；静态枚举、默认值留在 **form form** 即可。

## 19. SDK Issue 280 的复现逻辑 纯文字

未打补丁时 daemon 侧逻辑顺序大致为：

1. 收到 `fetch_parameter_options`  
2. SDK 在 `_get_dynamic_parameter_action` 中先 `try_get_trigger_event_handler`  
3. 对 Tool 提供方名调用 `TriggerFactory._get_entry`，找不到 Trigger 提供方  
4. 抛 `ValueError: Trigger provider xxx not found`  
5. 异常向上，**轮不到** `get_tool_cls` 实例化 `DynamicDeviceQueryTool`  

表现与「前端没请求」易混淆：前端有请求但 api 或 daemon 400，浏览器 options 仍为空。  

**验证是否 280**：看 daemon 日志是否有 Trigger provider not found 字样；若有且尚无 `_fetch_parameter_options called`，应先确认 `plugin_bootstrap.apply_sdk_patches()` 在 `main.py` 最前执行且 `.difypkg` 内含该文件。

## 21. generic_http 与 dynamic_device_query 对照实验

| 维度 | generic_http | dynamic_device_query |
|------|--------------|----------------------|
| 目的 | 任意 REST 调用 | 设备下拉加三类查询 |
| path 参数 | string 加 mixed | 无 path |

## 28. 十三个用户问题的原文脉络与回答摘录

本节把对话中反复出现的十三个技术问题，用「问题—我们当时怎么查—结论」再展开一遍，方便单独阅读本篇时不依赖聊天记录。

**问题 1：difypkg 里到底有什么？**  
我们执行 `dify plugin package` 后检查产物大小约数十 KB，解压可见 manifest、provider、tools 下全部 py 与 yaml，**不包含** Spring Boot 工程。动态参数相关代码必须在包内的 `dynamic_device_query.py` 与 `plugin_bootstrap.py` 中，否则装到服务器也不生效。

**问题 2：Python 插件项目干什么？**  
它运行在 daemon 分配的容器里，对工作流来说是「工具实现体」。配置期它响应 `_fetch_parameter_options`，运行期响应 `_invoke`。Spring 永远不会被 Dify 直接集成，只能通过 HTTP 被插件访问。

**问题 3：每个后端接口都要写一个工具吗？**  
不需要。FAQ 已验证 generic_http 可 GET/POST 五个 Device API。动态下拉是 UX 增强，不是 API 覆盖的必要条件。

**问题 4：插件升级会不会破坏工作流？**  
工具 name 不变则节点仍指向同一 identity；我们把版本升到 0.0.2 是为区分旧包，建议卸载后安装新包以免 daemon 缓存旧依赖。

**问题 5：能否用 Go 写插件？**  
官方 Runner 以 Python 为主，本文不涉及多语言。

**问题 6：通用 HTTP 能否解决所有问题？**  
运行期能。配置期不能自动把接口返回值变成另一参数的下拉。这是本次动态参数需求的出发点。

**问题 7：插件重名怎么办？**  
改 manifest 的 author 或 name；与动态参数无强关联。

**问题 8：多参数怎么传？**  
query_type、metric、limit 放在 form form，作为工具设置；device_id 放在 form llm，作为输入变量。执行时 daemon 合并 tool_configurations 与 tool_parameters 传入 `_invoke`。

**问题 9：能否下拉来自后端？**  
能。标准答案是 dynamic-select。本条是整个系列的中心。

**问题 10：只改 generic_http.yaml 够不够？**  
不够。且用户明确要求不修改 generic_http，故新建工具。

**问题 11：新建工具后仍不能选设备？**  
进入本篇核心排查：前端、SDK、网络三层。

**问题 12：请对照 dify 源码分析？**  
阅读 `dify/web/.../tool/panel.tsx` 与 `form-input-item.tsx`，确认第二个 ToolForm 缺陷。

**问题 13：写博客记录是否支持动态参数？**  
即本文。

### 29.1 现象甲 完全无 dynamic-options

| 观察点 | 记录 |

| curl 本机 | Spring 200 正常 |

```mermaid

    S[开始] --> L[获取设备列表]
    L --> E[结束或下游处理]
```

## 32. 打包产物与安装后文件关系

`.difypkg` 本质是压缩包。安装时 daemon 解压、创建隔离 Python 环境、`uv pip install -r requirements.txt`、根据 manifest 的 entrypoint 启动 `main`。  

若 requirements 仍为旧版 dify_plugin，则即使 zip 内含 dynamic_device_query.py，枚举类型仍可能不认识 dynamic-select，安装任务会报错或降级。  

**建议在安装日志中确认**：Successfully installed dify-plugin-0.9.x。

## 33. 与 Trigger 插件 dynamic-select 的对比学习

Trigger 文档示例中 repository 字段使用 dynamic-select，拉取逻辑在 Trigger 类 `_fetch_parameter_options` 中。前端对 Trigger 使用 `useTriggerPluginDynamicOptions`，在 `form-input-item` 里当 providerType 为 trigger 时走 query 而非 mutation。  

Tool 插件在工作流里使用 `useFetchDynamicOptions` mutation，在组件 mount 时触发。  

二者后端都汇聚到 `PluginParameterService.get_dynamic_select_options`，最终都到 daemon 的 `dynamic_select/fetch_parameter_options`。  

**差异在 UI 挂载与 panel 传参**，不在 Spring 或业务 API 设计。学习 Trigger 成功案例时，不要直接复制其 form 配置到 Tool 的 form form 区。

## 39. 深度排查案例 逐层打开黑盒

假设同事反馈：「动态设备查询装好了，设备下拉还是灰的。」按下列顺序带他做，每一步写下实际输出。

**层 0 需求确认**  
问清是「下拉没有选项」还是「选了设备运行失败」。前者走配置期链路，后者走 invoke 链路。本文主要针对前者。

**层 1 本机 Spring**  
在其开发机执行 curl device-select-options。若这里失败，先 `mvn spring-boot:run`，与 Dify 无关。我们实测 200 且三条 JSON，记录应一致。

**层 2 插件包内容**  
解压或检查构建机上的 `iot_device_http.difypkg` 是否含 `tools/dynamic_device_query.py`、`plugin_bootstrap.py`、`requirements.txt` 中 0.9.0 约束。缺一则重装包。

**层 3 控制台凭证**  
截图 spring_service_url。若是 localhost 而 daemon 在 K8s，判为根因。改为 `http://10.x.x.x:8080` 后重装凭证或重新保存。

**层 4 节点 UI 分区**  
截图节点面板。若 device_id 出现在「设置」折叠区而非「输入变量」，且 manifest 仍为 form form，判为前端 bug 命中。指导改 0.0.2 包或改 yaml 为 form llm。

**层 5 浏览器 Network**  
过滤 dynamic-options。无请求：停在层 4。有请求看 status 与 response body。400 记下 message 给 api 日志。200 且 options 长度 3 但 UI 空，查前端控制台 JS 错误，少见。

**层 6 daemon**  
`kubectl logs -f` 插件相关 pod。搜索 dynamic_select、fetch_parameter_options、Trigger provider。结合层 7。

**层 7 插件 stderr**  
远程安装时 stderr 在 daemon 聚合日志。必须有 `[dynamic_device_query] _fetch_parameter_options called`。有 called 后有拉取失败则把完整 URL 与异常贴给网络组查防火墙。

**层 8 仍不行**  
启用 debug 安装模式，本机 `python -m main` 连远程 debug.dify.ai，在本地 IDE 断点 `_fetch_parameter_options`。若本地断点能进、远程不能进，问题在安装包版本或补丁未生效。

以上八层可在值班手册中做成检查表，避免每次从 Spring 改代码开始。

## 43. 扩展阅读：dynamic-select 从平台 PR 到业务落地的认知路径

2025 年中 Dify 在参数实体层引入 DYNAMIC_SELECT，插件 SDK 随之在 0.9.x 暴露 `_fetch_parameter_options`。对我们 IoT 场景而言，这意味着「设备主数据在 Spring」「展示在 Dify 画布」可以通过官方扩展点连接，而不必在插件里写死三台设备。落地时却发现：**平台能力上线**与**某一类 UI 入口接满**之间存在时间差。工作流 Tool 节点的「设置」区就是未接满的典型。

从产品角度，用户自然希望设备列表与 Spring 保持同步：新增 device_004 后，下次打开下拉应四项。只要 `_fetch_parameter_options` 每次被调用时都 GET 最新列表，就能满足。静态 yaml select 需要改包重发，dynamic-select 只需刷新节点或重新 mount 表单项。我们选 dynamic-select 是为了 **运维上只改 Java**。

从安全角度，Spring 接口未对公网暴露，仅插件容器在内网访问，符合常见企业架构。不要在 Dify 侧重复实现设备 CRUD，保持 Spring 为唯一真相源。

从测试角度，建议维护三张用例表：配置期下拉、运行期 invoke、兜底 list_devices。回归时任选一张执行，避免只测运行而漏测配置。

## 44. 常见对话误解与纠正

**误解 A**：「Dify 不支持动态参数，因为下拉是空的。」  
**纠正**：在多数情况下是 UI 未请求或 SDK 中断，不是平台没有此类型。

**误解 B**：「Spring 接口返回了，Dify 就应该自动显示。」  
**纠正**：必须经过插件 Python 转换格式，且前端要发起 dynamic-options。

**误解 C**：「和 Trigger 一样配 form form 即可。」  
**纠正**：Tool 的 panel 结构与 Trigger 不同，不能照搬表单位置。

**误解 D**：「升级到 1.12 就够了。」  
**纠正**：版本满足最低要求仍可能踩 panel bug，需 form llm 或补丁。

**误解 E**：「debug 能 invoke 就说明下拉也会好。」  
**纠正**：invoke 与 fetch_parameter_options 是不同 dispatch 类型。

## 结语

**Dify 支持动态参数**，在插件领域即 **`dynamic-select` + `_fetch_parameter_options`**。  
我们要的「设备列表来自 Spring 接口」在架构上完全成立，但实现上必须满足：

1. Spring 提供选项 API，且地址对 **插件容器** 可达；  
2. 插件 SDK 0.9+ 与 280 补丁；  
3. 工作流 UI 能发起 `dynamic-options`——在 1.12 上对 Tool 插件 means **`form: llm`** 或修 panel。

本篇记录的失败并非「Dify 不支持」，而是 **配置分区与前端传参** 叠加 **SDK 行为** 导致的「看起来像没调后端」。按本文排查表逐层验证，可以把问题钉死在具体一层，避免在 Spring 或浏览器上空耗时间。
