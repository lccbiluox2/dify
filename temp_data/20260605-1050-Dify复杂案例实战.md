# Dify IoT 插件复杂案例实战：多工具链式工作流 + dynamic-select + base64 JSON + 上游变量引用 + 自定义参数

> 文档版本：2026-06-05  
> 插件版本：0.0.8  
> Dify 版本：v1.13+  
> 适用读者：需要基于 Dify 工作流编排 IoT 设备管控能力的开发者

## 1. 案例概述

### 1.1 要实现什么

在 Dify 工作流画布中搭建一个 **多工具链式工作流**，实现：

1. **获取 IP 列表**：调用 Spring Boot 后端，返回所有 IoT 设备关联的 IP 地址（含设备信息）
2. **级联设备管控**：基于上游 IP 列表的输出，通过变量引用将 IP 传入设备管控节点，执行 IP 封禁/解封等操作
3. **动态下拉选设备**：设备选择使用 `dynamic-select` 动态下拉，value 是 base64 编码的 JSON 富数据
4. **用户自定义输入**：支持手动输入备注 + JSON 格式的自定义复杂参数

### 1.2 工作流拓扑

```
┌──────────┐     ┌──────────────┐     ┌──────────────────┐     ┌──────────┐
│ 用户输入  │ ──▶ │  获取IP列表   │ ──▶ │  级联设备管控      │ ──▶ │   输出    │
│  (start)  │     │ (list_ips)   │     │ (cascading_action)│     │  (end)   │
└──────────┘     └──────────────┘     └──────────────────┘     └──────────┘
                                          ↑
                                          │ target_ip 引用上游节点 text 输出
```

### 1.3 涉及的技术栈

| 层 | 技术 | 说明 |
|----|------|------|
| 业务后端 | Spring Boot 3.x + Lombok | 提供 Mock 设备数据 + REST API |
| Dify 插件 | Python 3.12 + dify-plugin SDK | 连接 Dify 与 Spring Boot |
| Dify 平台 | Dify v1.13+（K8s 部署） | 工作流编排引擎 |
| 插件打包 | `dify plugin package` CLI | 生成 `.difypkg` 安装包 |

#### 4.4.3 POST /execute — 执行操作

```
请求: POST /api/cascading-device/execute
Content-Type: application/json

{
  "deviceInfo": "eyJkZXZpY2VJZCI6ImRldmljZV8wMDIiLCJk...",  // base64 JSON 原始值
  "targetIp": "192.168.1.201",
  "actionType": "ip_unblock",

  "customParams": "{\"blockParamType\":\"regular\",\"blockIPType\":\"attacker\"}"

```

**Java 端解码流程**：
1. `Base64.getDecoder().decode(deviceInfo)` → 字节数组
2. `new String(bytes, UTF_8)` → JSON 字符串
3. `objectMapper.readValue(json, CascadingDeviceInfo.class)` → 设备对象
4. 校验 targetIp 是否属于设备的 ips 列表
5. 校验 actionType 是否在 availableActions 中
6. 返回执行结果

**响应**：
```json

  "actionType": "ip_unblock",
  "actionLabel": "IP解封",
```

## 5. Dify 插件实现

### 5.1 工具一：获取IP列表（list_ips）

#### 5.1.1 工具定义（list_ips.yaml）

```yaml
identity:
  name: list_ips
  author: your-name
  label:
    zh_Hans: 获取IP列表
description:
  human:
    zh_Hans: "获取所有IoT设备关联的IP地址列表"
  llm: "Retrieve a list of all IP addresses across IoT devices."
parameters:
  - name: device_id
    type: string
    required: false
    form: form            # ← form:form 在设置区，支持 {x} 变量图标
    default: ""
```

**关键设计**：`form: form` 使参数出现在设置区（第二个 ToolForm），该区域支持 `{x}` 变量图标，可引用上游节点输出。

#### 5.1.2 工具实现（list_ips.py）

```python
class ListIpsTool(Tool):
    LIST_IPS_PATH = "/api/cascading-device/all-ips"

    def _invoke(self, tool_parameters):
        # 1. 从凭证获取 Spring Boot 地址
        spring_url = self.runtime.credentials.get("spring_service_url", "").rstrip("/")
        device_id = (tool_parameters.get("device_id") or "").strip()

        # 2. 调用 Spring Boot 接口
        url = f"{spring_url}{self.LIST_IPS_PATH}"
        params = {"deviceId": device_id} if device_id else {}
        response = requests.get(url, params=params, timeout=15)
        ips = response.json()

        # 3. 格式化输出（两种消息类型）
        yield self.create_text_message("共发现 N 个 IP 地址：\n• IP: xxx\n  设备: ...")
        yield self.create_json_message({"ips": ips, "total": len(ips)})
```

**输出说明**：
- `text_message`：人类可读的格式化文本，下游节点可通过 `#节点ID.text#` 引用
- `json_message`：结构化 JSON，下游节点可通过 `#节点ID.json.xxx#` 引用具体字段

### 5.2 工具二：级联设备管控（cascading_device_action）

#### 5.2.1 工具定义（cascading_device_action.yaml）

```yaml
parameters:
  # 参数1：dynamic-select 动态下拉（唯一的配置期参数）
  - name: device_info
    type: dynamic-select
    required: true
    form: llm              # ← form:llm 进入输入变量区

  # 参数2：目标IP（string，支持上游变量引用）
  - name: target_ip
    type: string
    required: true
    form: form              # ← form:form 在设置区，有 {x} 变量图标

  # 参数3：操作类型（静态下拉）
  - name: action_type
    type: select
    required: true
    form: form
    default: ip_block
    options:
      - value: ip_block      # IP封禁
      - value: ip_unblock    # IP解封
      - value: domain_block  # 域名封禁
      - value: domain_unblock# 域名解封
      - value: custom_rule   # 自定义规则

  # 参数4：用户输入（可选字符串）
  - name: user_input
    type: string
    required: false
    form: form

  # 参数5：自定义参数 JSON（可选复杂参数）
  - name: custom_params
    type: string
    required: false
    form: form
```

**参数分区设计原理**：

| 参数 | form | 出现在画布哪个区域 | 变量图标 |
|------|------|--------------------|----------|
| device_info | `llm` | 输入变量区（第一个 ToolForm） | 无（dynamic-select 不支持） |
| target_ip | `form` | 设置区（第二个 ToolForm） | 有 `{x}` |
| action_type | `form` | 设置区 | 有 `{x}` |
| user_input | `form` | 设置区 | 有 `{x}` |
| custom_params | `form` | 设置区 | 有 `{x}` |

#### 5.2.2 工具实现（cascading_device_action.py）

**核心流程**：

```
1. 读取 tool_parameters
2. 解析 custom_params JSON（容错处理）
3. 解码 device_info base64 JSON → 提取 deviceId, ips, actions
4. 校验 target_ip 是否属于设备 IP 列表
5. 校验 action_type 是否在设备支持的操作列表中
6. POST /api/cascading-device/execute 执行操作
7. 输出 text_message + json_message
```

**Python → Java 的请求体**：

```python
payload = {
    "deviceInfo": raw_device_info,     # base64 原始值，Java 端解码
    "targetIp": target_ip,             # 具体 IP 地址
    "actionType": action_type,         # 操作类型
    "userInput": user_input,           # 用户文本输入
    "customParams": json.dumps(custom_params, ensure_ascii=False),  # JSON 字符串
}
```

## 6. 工作流画布搭建（详细步骤）

### 6.1 第一步：创建工作流

1. 登录 Dify Web 界面（如 `http://10.20.183.170:30080`）
2. 进入「工作室」→「创建应用」→ 选择「工作流」
3. 命名为 `IoT设备管控工作流`

### 6.2 第二步：添加「开始」节点

默认已有「用户输入」开始节点，无需额外配置。

### 6.3 第三步：添加「获取IP列表」节点

1. 点击「+」添加节点 → 选择「工具」→ 找到 **IoT设备通用网关** → 选择 **获取IP列表**
2. 将「开始」节点的输出连接到「获取IP列表」节点的输入
3. **参数配置**：
   - `device_id`：留空（获取所有设备的 IP）或输入特定设备 ID

### 6.4 第四步：添加「级联设备管控」节点

1. 点击「+」添加节点 → 选择「工具」→ **IoT设备通用网关** → 选择 **级联设备管控**
2. 将「获取IP列表」节点的输出连接到「级联设备管控」节点的输入
3. **参数配置**：

| 参数 | 配置方式 | 值 |
|------|----------|------|
| `device_info`（设备） | 从动态下拉选择 | 选择如「卧室智能灯泡 [online] \| 3个IP \| 4个操作」 |
| `target_ip`（目标IP） | 引用上游变量 | `#获取IP列表.text#` |
| `action_type`（操作类型） | 静态选择 | `IP解封` |
| `user_input`（用户输入） | 手动输入 | `联动设备管控-手动输入` |
| `custom_params`（自定义参数） | JSON 字符串 | `{"blockParamType":"regular","blockIPType":"attacker","customParam":"single"}` |

**⚠️ 重要：target_ip 的变量引用写法**

在画布上，点击 `target_ip` 输入框右侧的 `{x}` 变量图标 → 选择「获取IP列表」节点 → 选择 `text` 输出。

Dify 会自动将其转换为 `#节点ID.text#` 格式（如 `#1780627507733.text#`）。

### 6.5 第五步：添加「输出」节点

1. 添加「结束」节点
2. 添加输出变量 `text`，引用 `#级联设备管控.text#`
3. 将「级联设备管控」的输出连接到「输出」节点

### 6.6 连线检查

最终画布连线应为：

```
开始 → 获取IP列表 → 级联设备管控 → 输出
```

### 7.3 常见报错：targetIp 不是有效 IP

**报错信息**：
```
IP 共发现 7 个 IP 地址：• IP: 10.0.0.100 ... 不属于设备 device_002
```

**原因**：`target_ip` 引用了上游 `list_ips` 工具的 `text` 输出，该输出是格式化的人类可读文本（包含所有 IP 的描述），不是一个纯净的 IP 地址。

**解决方案**：需要在上游节点和管控节点之间增加一个「代码节点」或「HTTP节点」，从 IP 列表中提取单个 IP 地址后再传入。具体做法：

1. 在「获取IP列表」和「级联设备管控」之间插入一个**代码节点**
2. 代码节点输入引用 `#获取IP列表.json#`（JSON 结构化输出）
3. 代码内容示例：
   ```javascript
   function main(args) {
     const ips = args.ips || [];
     // 找到属于 device_002 的第一个 IP
     const match = ips.find(ip => ip.deviceId === 'device_002');
     return { target_ip: match ? match.ipAddress : '' };
   }
   ```
4. 管控节点的 `target_ip` 改为引用 `#代码节点.target_ip#`

## 9. 关键设计决策与技术限制

### 9.1 为什么 dynamic-select 不能级联？

Dify 的 `dynamic-select` 存在 **5 项架构限制**：

1. `_fetch_parameter_options` 钩子只接收 `parameter: str`，不知道其他参数值
2. 后端 API 不传递表单其他参数状态
3. 前端 `extraParams` 始终为 `undefined`
4. `handleValueChange` 只更新当前参数值
5. dynamic-select 的值强制为 `constant`，不支持变量引用

**结论**：无法实现「选了设备 A → IP 下拉只显示 A 的 IP」这种级联效果。

### 9.2 base64 JSON 富编码的设计动机

由于 dynamic-select 无法级联，将设备的 **完整上下文信息**（deviceId、IPs、availableActions）序列化为 JSON → base64 编码，作为下拉选项的 value。这样：

- **配置期**：用户从下拉选择设备，看到友好的 label（设备名 + IP数 + 操作数）
- **运行期**：Python 和 Java 解码 base64 JSON，获得完整的设备信息用于校验和执行

### 9.3 form: llm vs form: form 的区别

| 属性 | `form: llm` | `form: form` |
|------|-------------|--------------|
| 画布位置 | 输入变量区（第一个 ToolForm） | 设置区（第二个 ToolForm） |
| `{x}` 变量图标 | 不一定显示 | 始终显示 |
| 适合场景 | dynamic-select、LLM 自动填充参数 | 用户手动配置 + 引用上游变量 |

### 9.4 custom_params 容错机制

| 用户输入 | Python 解析结果 | Java 解析结果 |
|----------|----------------|---------------|
| 空字符串 | `{}` | `null` → 不传递 |
| `{"a":1}` | `{"a": 1}` | `{"a": 1}` |
| `不是JSON` | `{"_raw": "不是JSON"}` | 解析失败，设为 `null` |
| `[1,2,3]`（数组） | `{"_raw": "[1,2,3]"}` | 正常解析为 Map |
