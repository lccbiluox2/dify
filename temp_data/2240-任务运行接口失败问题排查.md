# Dify 任务运行接口失败问题排查

## 一、问题概述

在实现 Dify 已发布工作流运行功能时，遇到了一系列认证和路由问题，导致无法成功调用 `/v1/workflows/run` 接口。

---

## 二、问题现象与排查过程

### 2.1 问题一：appCode 获取方式不确定

#### 现象描述
- 最初使用硬编码的 appCode：`olZeP6njJZMHZqHr`
- 不确定 appCode 应该从哪里获取

#### 排查过程
1. **查看接口文档**：`2240-任务运行接口.md`
2. **发现 appCode 来源**：从应用详情接口的 `site` 字段中获取
   ```json
   {
     "site": {
       "access_token": "olZeP6njJZMHZqHr",
       "code": "olZeP6njJZMHZqHr"
     }
   }
   ```

#### 解决方案
1. 在 `AppClient` 中新增 `getAppDetailRaw()` 方法返回原始 JSON
2. 在 `AppService` 中使用 Hutool HTTP 直接调用（绕过 Feign 解码问题）
3. 在测试中手动解析 JSON，提取 `site.access_token` 或 `site.code`

### 2.2 问题二：Feign 返回 String 类型解码失败

#### 现象描述
```
feign.FeignException: Cannot deserialize value of type `java.lang.String` 
from Object value (token `JsonToken.START_OBJECT`)
```

#### 原因分析
Feign 的默认解码器会尝试将 JSON 响应反序列化为指定的返回类型。当返回类型为 `String` 时，Jackson 期望接收纯字符串，但实际收到的是 JSON 对象（`JsonToken.START_OBJECT`），导致反序列化失败。

#### 解决方案
**不使用 Feign 客户端**，改为在 Service 层直接使用 Hutool HTTP：
```java
public String getAppDetailRaw(String appId) {
    cn.hutool.http.HttpResponse response = cn.hutool.http.HttpRequest.get(url)
        .header("x-csrf-token", apiProperties.getCsrfToken())
        .header("Cookie", apiProperties.getCookie())
        .execute();
    return response.body();
}
```

### 2.3 问题三：AppResponse DTO 字段不完整

#### 现象描述
```
feign.FeignException: Unrecognized field "tracing" (class com.dify.api.dto.response.AppResponse)
feign.FeignException: Unrecognized field "description" (class com.dify.api.dto.response.AppResponse$SiteInfo)
```

#### 原因分析
服务端返回的 JSON 包含多个字段，但 DTO 中未定义，Jackson 反序列化时抛出 `UnrecognizedPropertyException`。

#### 解决方案
补充 `AppResponse` 和 `SiteInfo` 中缺失的字段：
```java
// AppResponse.java
private Object tracing;
private String apiBaseUrl;
private Object maxActiveRequests;
private List<Object> deletedTools;
private String accessMode;
private String iconUrl;
private SiteInfo site;

// SiteInfo.java
private String description;
private String defaultLanguage;
private String chatColorTheme;
// ... 其他字段
```

### 2.4 问题四：x-app-passport 传递错误

#### 现象描述
```
HTTP Error: 401 - {"code":"unauthorized","message":"App token is missing.","status":401}
```

#### 原因分析
最初将完整的 Cookie 字符串传递给了 `x-app-passport` 请求头：
```java
.header("x-app-passport", apiProperties.getCookie())  // ❌ 错误
```

但服务端期望的是单独的 `access_token` 值。

#### Cookie 格式
```
csrf_token=eyJhbGciOiJIUzI1NiIs...; refresh_token=c71341f6bd46dc9c...; access_token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

#### 解决方案
从 Cookie 字符串中提取 `access_token`：
```java
private String extractAccessToken(String cookie) {
    String[] cookies = cookie.split(";");
    for (String c : cookies) {
        String trimmed = c.trim();
        if (trimmed.startsWith("access_token=")) {
            return trimmed.substring("access_token=".length());
        }
    }
    return null;
}

// 使用
String accessToken = extractAccessToken(apiProperties.getCookie());
.header("x-app-passport", accessToken)  // ✅ 正确
```

### 2.5 问题五：API 路径错误

#### 现象描述
```
HTTP Error: 404 - {"code":"not_found","message":"You have requested this URI [/api/workflows/run] 
but did you mean /api/workflows/run or /v1/workflows/run or /v1/workflows/run/<string:workflow_run_id> ?"}
```

#### 原因分析
使用了错误的 API 路径：`/api/workflows/run`

#### Dify API 路径说明
Dify 有两套 API：

1. **Console API**（控制台管理接口）
   - 路径：`/console/api/*`
   - 认证：Cookie + CSRF Token
   - 示例：`/console/api/apps/{app_id}/workflows/draft/run`（运行草稿）

2. **Service API**（服务调用接口）
   - 路径：`/v1/*`
   - 认证：`x-app-passport` (access_token)
   - 示例：`/v1/workflows/run`（运行已发布的工作流）

#### 解决方案
```java
// 修改前
String url = apiProperties.getBaseUrl() + "/api/workflows/run";  // ❌ 错误

// 修改后
String url = apiProperties.getBaseUrl() + "/v1/workflows/run";  // ✅ 正确
```

## 五、关键技术点

### 5.1 Dify 认证机制

#### Console API 认证
```
Header: x-csrf-token: <csrf_token>
Header: Cookie: csrf_token=...; access_token=...; refresh_token=...
```

#### Service API 认证
```
Header: x-app-code: <appCode>
Header: x-app-passport: <access_token>
Header: Cookie: <完整Cookie>
```

### 5.2 appCode 获取流程

```
1. 调用 GET /console/api/apps?page=1&limit=10&name=Workflow_C
   → 获取应用列表，得到 appId

2. 调用 GET /console/api/apps/{app_id}
   → 获取应用详情，解析 JSON

3. 从 response.site.access_token 或 response.site.code
   → 提取 appCode
```

### 5.3 access_token 提取逻辑

## 六、经验总结

### 6.1 Dify API 集成注意事项

1. **区分 API 类型**：
   - Console API 用于管理操作（需要 CSRF Token）
   - Service API 用于业务调用（需要 access_token）

2. **认证头规范**：
   - 不要混淆 Cookie、CSRF Token、access_token 的用途
   - `x-app-passport` 必须是纯 access_token 值

3. **Feign 使用限制**：
   - 返回 `String` 类型会触发解码问题
   - 需要原始响应时使用 Hutool HTTP 或其他工具

4. **DTO 设计原则**：
   - 必须覆盖服务端返回的所有字段
   - 不确定类型的字段可使用 `Object` 或 `Map`

### 6.2 排查技巧

1. **查看服务端提示**：404 错误中服务端会给出正确的路径建议
2. **打印完整请求头**：便于定位认证问题
3. **保存原始 JSON**：手动解析可以绕过 Jackson 的限制
4. **对比接口文档**：curl 示例中的请求头是重要参考
